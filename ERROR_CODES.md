# Sentinel CI: Error & Analysis Reference 📖

This document helps developers understand the findings generated by **Sentinel CI** (AI Code Guardian engine) and how to troubleshoot common issues.

---

## 🛡️ Finding Classifications

Sentinel CI categorizes issues based on their impact and the confidence of the detection engine.

### 🔴 ERROR (Critical)
**Definition:** High-probability security vulnerabilities or critical architectural violations.
- **Impact:** Likely exploitable. Direct threat to application security or stability.
- **CI/CD Behavior:** By default, these findings **fail the build** and block PR merges.
- **Action Required:** Immediate fix required before merging.

### 🟡 WARNING (Medium)
**Definition:** Potential security concerns, complex code smells, or high-risk patterns.
- **Impact:** May be exploitable depending on context. Represents technical debt.
- **CI/CD Behavior:** Reported in the PR comment; merge blocking depends on your `.sentinel-ci.yml` (or legacy `.ai-guardian.yml`) `block_on_severity` settings.
- **Action Required:** Review suggested fixes and address if applicable.

### 🔵 INFO (Low/Best Practice)
**Definition:** General secure coding recommendations, naming convention warnings, or performance tips.
- **Impact:** Low risk. Primarily aimed at maintaining long-term code quality.
- **CI/CD Behavior:** Reported for developer awareness.
- **Action Required:** Adopt as a best practice during follow-up refactors.

## ❓ Troubleshooting

### 1. "No findings reported, but I know there's a bug"
Sentinel CI focuses on **patterns** and **context**. If a specific bug doesn't match a known vulnerability pattern or architectural rule, it may be missed. We recommend using it alongside dynamic testing (DAST).

### 2. "False Positive: The AI is wrong"
AI is used to explain and filter findings. If you believe a finding is incorrect:
- Check the rule ID in the report.
- You can ignore specific lines by adding a comment: `# nosemgrep` (for security rules).
- Adjust your `.sentinel-ci.yml` to exclude specific paths.

### 3. "GitHub Action failed with exit code 1"
This is expected behavior when **ERROR** level findings are detected (or **WARNING**, if you enabled strict blocking). Review the PR comment from the bot, apply the suggested fixes, and push an update to clear the check.

## 📊 Understanding the Risk Score
The dashboard summary in your PR includes a **Risk Score (0-100)**:
- **0-10:** Healthy code.
- **11-40:** Minor technical debt or low-risk findings.
- **41-70:** Significant security or quality concerns.
- **71+:** High-risk PR; requires deep manual audit.
